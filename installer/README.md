# Prezo Windows Installer

Per-user, no-admin installer that registers Prezo as a PowerPoint task pane add-in via Office's `TrustedCatalogs` registry mechanism. Replaces the manual sideload + Trust Center flow.

## End-user flow

1. Download `PrezoSetup-x.y.z.exe` from the Prezo site.
2. Double-click to run. SmartScreen will warn ("unidentified developer") because the binary is unsigned. Click **More info -> Run anyway**.
3. Welcome -> Install -> Finish. No admin prompt.
4. Restart PowerPoint. Prezo appears under the **Home** tab.

## Layout

```
installer/
  build.ps1              One-command build script (also discovers manifests)
  README.md              This file
  src/
    prezo-installer.iss  Main script (orchestrates includes)
    config.iss           App metadata: AppId, publisher, Office version, etc.
    files.iss            [Files] for static assets (icon)
    manifests.iss        AUTO-GENERATED by build.ps1 — do not edit
    code.iss             [Code] — PowerPoint detection & restart hint
  assets/
    manifest*.xml        Staged from frontend-addin/manifest at build time
    prezo-icon.ico       Optional — installer + uninstall icon
  dist/                  Build output (gitignored)
```

Each `.iss` file owns one concern. `config.iss` is the only file you should ever edit between releases for routine changes. **Never edit `manifests.iss`** — `build.ps1` overwrites it on every run.

### Adding a new add-in surface

Drop a new `manifest-<something>.xml` file into `frontend-addin/manifest/` and rerun `.\build.ps1`. The build script discovers every `manifest*.xml` file there, copies them into `installer/assets/`, and emits a fresh `manifests.iss` with one `[Files]` line and one `[Registry]` line per manifest. No edits to the installer source needed.

## Prerequisites

- Windows
- [Inno Setup 6](https://jrsoftware.org/isdl.php) (free)
- PowerShell 5.1 or newer (built into Windows)

## Build

From the repo root or anywhere:

```powershell
cd installer
.\build.ps1
```

Output: `installer/dist/PrezoSetup-<version>.exe`

The version is read from `<Version>` in `frontend-addin/manifest/manifest.xml`. To override:

```powershell
.\build.ps1 -Version 1.2.0
```

If ISCC is not auto-detected:

```powershell
.\build.ps1 -Iscc "C:\Path\To\Inno Setup 6\ISCC.exe"
```

## How it works

The installer drops every `manifest*.xml` into `%LOCALAPPDATA%\Prezo\Catalog\` and writes one REG_SZ value per manifest under `HKCU\Software\Microsoft\Office\16.0\Wef\Developer`. The value name is the manifest's `<Id>` GUID; the value data is the absolute path to that manifest.

This is the same mechanism used by Microsoft's `office-addin-dev-settings` CLI and PowerPoint's "Upload My Add-In" dialog, so it works reliably across Office desktop builds. Unlike the older TrustedCatalogs route, it accepts plain local file paths with no UNC share or HTTP URL required, which makes it usable from a per-user install with no admin rights.

On next PowerPoint launch, Office reads the Developer key, loads each manifest, and:

- Surfaces the host taskpane add-in's ribbon button (from `VersionOverrides`) on the Home tab.
- Makes the content add-in available via Insert -> Add-ins (it embeds directly into a slide when the user picks it).

The Prezo web apps themselves stay on Railway. The installer ships only the manifest XMLs and a few hundred bytes of registry metadata, so users get fresh React code on every PowerPoint launch with no installer update needed.

## Test

After a build:

1. Run `installer\dist\PrezoSetup-<version>.exe` on a clean Windows machine (or VM).
2. Verify install completes without UAC prompt.
3. Open `regedit` and confirm `HKCU\Software\Microsoft\Office\16.0\Wef\Developer` has one REG_SZ value per manifest, each pointing at a `%LOCALAPPDATA%\Prezo\Catalog\manifest*.xml` path.
4. Open PowerPoint -> Home tab. Prezo button should be visible.
5. Click Prezo Host -> task pane should load the Railway app.
6. Insert -> Add-ins -> Developer Add-Ins -> the content app entry should appear and embed into a slide when chosen.
7. Uninstall via Settings -> Apps. Confirm the manifest values under `Wef\Developer` and the `%LOCALAPPDATA%\Prezo` folder are both gone.

## Limits and known issues

- **No code signing.** SmartScreen warning on first run is expected. Acceptable for the internal phase; add an EV certificate before the public launch.
- **Windows only.** Mac users still need a separate `.pkg` installer (not yet built).
- **Locked-down corporate environments.** Some Group Policies block writes to `HKCU\Software\Microsoft\Office\16.0\WEF`. The installer will succeed but Prezo will not appear. There is no client-side fix; ask client IT to deploy via M365 Integrated Apps.
- **Office Web.** PowerPoint in the browser cannot use installer-based registration. AppSource or admin Centralized Deployment are the only options there.

## Release checklist

1. Bump `<Version>` in `frontend-addin/manifest/manifest.xml`.
2. Run `.\build.ps1` from `installer/`.
3. Smoke-test the resulting `.exe` on a clean machine (see Test section).
4. Upload to the Prezo download page.
